SECTION 01

How it works — the 3 promises

Provenfolio is a system for building a portfolio that never oversells. Behind the scenes, here is what happens.

To protect your credibility, three promises are built into the system.

  1. Every number is a real measurement. Commit counts, days of development, language ratios — every figure shown on the site is measured automatically from your git history by a program, and nothing else. The AI has no structural way to inflate a number.
  2. Every claim links to evidence. A highlight like "dramatically faster startup" always carries a link to its proof (a commit, a file, or a pull request). Claims without evidence never make it onto the site.
  3. You are the final owner of the words. The AI writes the first draft, but text you have edited is never silently overwritten by a re-analysis (your edits are detected automatically).
TIP

The data/ folder = your property. Your profile, project data, and screenshots all live in here. Everything else is just the "engine" (the tool) — as long as you have data/, the site can be rebuilt at any time.

SECTION 03

Adding a project — the operation you'll use most

Just hand over a GitHub repository URL, and everything runs on its own: analysis → draft writing → translation.

SAY THIS TO CLAUDE "/analyze https://github.com/(your-account)/(repo-name)"

"Add this project to my portfolio: (URL)" works too. For work that isn't on GitHub, just say "I want to add a project manually."

What "your turn" asks you

You are only asked for things a machine cannot know — things only you know. You can also fill these in later from Studio's Intake tab.

  • Category — product / client work / open source / hobby / learning: which is it?
  • Role — did you build it alone? If it was a team, what was your part?
  • Motivation — why did you build it? What were you trying to solve?
  • Demo links — a URL where people can see it working (app stores and videos count)
  • Outcomes you can put numbers on — users, downloads, speed improvements, and so on (if any)
TIP

Projects you add never appear on the internet until you publish. Feel free to add as many as you like, see how they turn out in Studio, and publish when you're ready.

SECTION 05

Editing text and keeping translations in sync

The AI's draft is a starting point. It only becomes your portfolio once it's in your own words. There are two ways:

  • Studio's Prose tab — edit the text boxes directly yourself (saved on the spot)
  • Ask Claude — when you'd rather work through it in conversation
SAY THIS TO CLAUDE "/edit (project-name) — make the summary more concise and lead with the results"

When you fix the original text, the translated version is brought along automatically. "Just redo the translation" works too.

IMPORTANT

Text you have edited is protected. Re-analysis will not overwrite it. Only if you want the AI to rewrite it should you say so explicitly.

SECTION 08

Preview and publish

Always preview before publishing

SAY THIS TO CLAUDE "/update-site"

It checks the data, generates the site, and opens it in your browser. To just look on your own, npm run dev → localhost:4321 works too.

Publish

SAY THIS TO CLAUDE "/publish"

It runs the completeness check → build → a "really publish?" confirmation → publish. Nothing ever goes live without your confirmation.

In some environments, a security restriction stops Claude from running just the final deploy step. If that happens, type this one line into your own terminal, inside the Provenfolio folder:

TERMINAL — deploy command
npx wrangler pages deploy site/dist --project-name (your-project-name) --branch main

Your public URL updates within seconds. You can publish again as many times as you like (each publish replaces the whole site).

SECTION 09

Weekly auto-updates (set it up once, then forget it)

Once set up, every Friday at 21:00 UTC (adjust the schedule in refresh.yml if you like) the whole cycle runs fully automatically: re-measure your development numbers → regenerate the site → publish. You just keep coding the way you always do, and your portfolio's numbers stay fresh on their own.

First-time setup (10 minutes, once)

This step hands a "key" (an API token) over to GitHub, so this is the one part you do with your own hands (Claude is not allowed to handle credentials). The steps assume you publish with Cloudflare Pages.

  1. Copy your Cloudflare account IDLog in at dash.cloudflare.com → in the URL, dash.cloudflare.com/(this part) is your account ID
  2. Create an API tokenProfile icon (top right) → My Profile → API Tokens → Create Token → Create Custom Token. Under Permissions add one row: "Account / Cloudflare Pages / Edit" → Create. Copy the token right there (it will never be shown again!)
  3. Register two Secrets on GitHubYour instance repository (on GitHub) → Settings → Secrets and variables → Actions → the Secrets tab → New repository secret: add CLOUDFLARE_API_TOKEN (the value from step 2) and CLOUDFLARE_ACCOUNT_ID (the value from step 1)
  4. Register one VariableOn the same screen, the Variables tab → New repository variable: CF_PAGES_PROJECT = your Pages project name
  5. Test itYour repository's Actions tab → "refresh" → Run workflow. A green check means you're all set
TIP

Only if you analyze private repositories do you also need the Secret GH_TOKEN (a GitHub Personal Access Token with repo read access). If all your repos are public, you don't need it.

SECTION 10

Receiving engine updates

Provenfolio itself (the engine) keeps evolving in its distribution repository. When new features and themes come out, you can pull them in — and your data/ (projects, text, photos) is structurally impossible to break in the process.

① With the Studio button (easiest)

  1. Press "Engine update" at the bottom left of Studio
  2. "Check for updates" → if anything is new, a list of changes appears
  3. "Update now" → fetch, merge, and rebuild all run automaticallyLogs stream on screen. If the merge fails, everything is rolled back automatically
  4. When it finishes, restart StudioIt starts running on the new engine

② Ask Claude

SAY THIS TO CLAUDE "Pull in the engine updates"

It does the whole run: fetch → merge → dependency updates → data migration → validation → build check. It's also the way to sort things out (update conflicts and such) when the button fails.

SECTION 12

Troubleshooting Q&A

Q. I changed the theme but the site looks the same

Design changes take effect the next time the site is generated and published. Say "run /update-site" to check the preview, then "/publish" to go live. If a preview server was left running, it needs a restart ("restart the dev server").

Q. It won't start — something about "the port is in use"

SAY THIS TO CLAUDE"Stop whatever is using port 4600 and start Studio again"

Q. A build or command stopped with a red error

SAY THIS TO CLAUDE"Run npm run validate and fix the errors"

Pasting the error message as-is is the fastest route.

Q. An engine update stopped partway through

The reason is in Studio's log. If the merge failed, everything was rolled back automatically; if a later step stopped (dependency install or the build), the update is applied but not yet verified. Either way, ask Claude to "pull in the engine updates; it just failed in Studio" and it will solve it, root cause and all.

Q. The DESIGN button at the bottom right of the site disappeared

It hides automatically when only one design would be offered to visitors — the default theme alone (that's intentional). In Studio's Site design panel, check at least one theme other than the default.

Q. I want to unpublish, or roll back a bad publish

Each publish replaces the whole site, so fixing things and publishing again overwrites it. To take the site down entirely: Cloudflare dashboard → Workers & Pages → your project → Settings → Delete.

Q. How much can I safely leave to Claude?

All of it. That said, three things are designed to always require your confirmation or your own hands: ① publishing to the internet (/publish), ② pushing to your repository, and ③ entering passwords or API tokens (Claude cannot do this for you).

SECTION 13

Mini glossary

Repository
A storage box for a program, kept together with its whole history of changes. GitHub is a service that hosts them
Commit
A saved checkpoint that says "done up to here". Provenfolio counts these to measure your track record
Build
Baking your data into a site (a bundle of HTML). You can do it locally, free, as many times as you like
Deploy / publish
Placing the baked site on the internet. Only then can the world see it
Preview
Looking at the site privately before publishing. Runs on localhost (inside your own PC)
Theme
A complete look for the site (colors, typefaces, layout). Swappable like a change of outfit
Intake
The form that collects what only you know about a project (role, motivation, and so on)
Case study
The write-up for one project. Auto-drafted in a problem → solution → results structure
Completeness score
How finished a project page is (out of 100). It also tells you what's still missing
API token
A spare key for operating a service. Treat it like a password — never hand it to a person or an AI (GitHub Secrets is its proper home)
Cloudflare Pages
A free hosting service (a place to put your site). The default publishing destination
Engine / instance
Engine = the Provenfolio tool itself (the distribution repository). Instance = your own private copy of it, data/ included
